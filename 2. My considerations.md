	What we have: a hand-rolled in-memory cache.
	Here's exactly how it works:
	```
	Request comes in
	       ↓
	Build cache key = cacheKey + url + JSON(params) + JSON(options)
	       ↓
	Is there a tempCache[key] entry AND is it still valid (within timer)?
	       ↓                              ↓
	YES → return cached response     NO → hit Axios, store response in tempCache
	```
	The `tempCache` is just a plain JavaScript object sitting in module memory:
	```
	const tempCache = {};  // lives in RAM, dies on page refresh
	```
It also has a `BroadcastChannel` so that if one browser tab clears cache, other tabs hear about it too — which is a thoughtful touch.

### Where our cache is good

It works for the basic case — same URL + same params within the timer window returns instantly without hitting the network. The BroadcastChannel cross-tab invalidation is genuinely clever and something React Query doesn't do out of the box.

### Where your cache falls short vs TanStack Query

Here's the honest comparison across every dimension that matters for Lens:

| |Your cache|TanStack Query|
|---|---|---|
|**Stale data while revalidating**|No — either cached or fresh, nothing in between|Yes — shows stale data instantly, fetches fresh in background|
|**Background refetch**|No|Yes — refetches on window focus, reconnect, interval|
|**Deduplication**|No — two components requesting same URL fire two requests|Yes — identical in-flight requests are merged into one|
|**Loading / error states**|You manage manually in each component|Built in — `isLoading`, `isError`, `isFetching` per query|
|**Cache invalidation**|Manual `clearCache()` call with key strings|`queryClient.invalidateQueries('key')` — triggers refetch automatically|
|**Persistence across pages**|No — `tempCache` is module-scoped, resets on full reload|Optional — `persistQueryClient` plugin for localStorage|
|**Garbage collection**|No — stale entries sit in `tempCache` forever|Yes — unused cache entries cleaned up automatically|
|**DevTools**|None|Full devtools panel showing every query, its state, and cache contents|
|**Request cancellation**|Manual `axios.CancelToken` per component|Automatic via `AbortController`|
|**Retry on failure**|None|Built in — retries failed requests with exponential backoff|
|**Parallel queries**|Manual `Promise.all`|`useQueries` hook handles it declaratively|
### The specific problem this causes for Lens LCP

The most damaging missing feature is **request deduplication**. Look at your `BillingSummaryRevamp` — it renders multiple `Widget` components simultaneously. If two widgets both call `GetCostSummary` with the same payload, your current cache doesn't help because both requests fire before either one resolves and stores a result. They race to the same endpoint and both wait for a response.

TanStack Query deduplicates at the in-flight level — the second component asking for the same data just subscribes to the already-in-flight promise. One network request, two components satisfied.

Your cache is opt-in per service method. Most of your service methods don't pass a `cache` object, which means they bypass `fetchWithCache` entirely and hit the network every single time. The cache only works where someone remembered to add it.

You already have `@tanstack/react-query` in your `package.json` — it's installed but not being used for these data fetches. You're paying the bundle cost without getting the benefit.
