# Problem

### SERVER METRICS

1. TTFB - (Time To First Byte) 
	How long before the server sends back the very first byte of HTML. Measures server speed - Your DB queries, auth, rendering on the backend.

### PAINT METRICS - WHAT THE USER SEES

1. FCP - (First Contentful Paint)
	When the user first sees anything on screen - a loading spinner, a navbar, any text or any image. "The page is alive."

2. LCP - (Largest Contentful Paint)
	When the biggest visible element (usually a chart, table, or hero image) is fully painted. This is what Google and users judge as "the page loaded."

3. SI - (Speed Index)
	How quickly the visible area fills up over time - scored as an averate. A page that paints gradually scores worse than one that paints all at once.

### INTERACTIVITY METRICS - CAN THE USER ACTUALLY USE IT

1. **TBT - (Total Blocking Time)** 
	Total time the main thread was "blocked" by long JS tasks (tasks > 50ms). Hight TBT = clicking feels frozen. Low TBT = JS bundle is lean.
	
2. **TTI - (Time to Interactive)**
	When the page is fully interactive - no long tasks queued, network quite, user can click and get response. TTI > LCP usually means backgroud JS running after paint.
	
3. **Web Vitals Bands - (Google's thresholds)**
	- Good : LCP < 2.5s. 
	- Needs Improvement : 2.5 - 4s
	- Poor : >4s.

### Where exactly is it lacking

The problem is the gap between FCP and LCP — 3.28 seconds of median dead time where the skeleton is visible but the charts haven't arrived yet. Since TBT is low, the browser's main thread is free during this time. So what's it waiting for?

The most probable culprit is serial API fetching — your dashboards likely do something like: mount the component → fetch config → then fetch data → then render the chart. Each arrow is a round-trip. If three fetches happen one after another instead of simultaneously, that's 3× a single fetch's latency sitting between FCP and LCP.
